QuantGym 模拟面试全面优化方案
================================================================

目标：把当前「单题一次性作答 → 打分 → 下一题」的练习器，升级为接近真人
面试的多轮对话式模拟，同时强化 AI 反馈深度、语音/多模态体验和题目覆盖。

交付定位：方案文档（先确认方向，再改代码）。对标方向：通用 / 全覆盖。
核心交互：多轮对话式追问（作为新的「真实面试模式 / live」，保留现有单题
流程作为「练习模式 / practice」）。

本版已并入与你逐项确认的设计决策（见第 0 节）；正文与路线图均按这些决策
调整。


----------------------------------------------------------------
0. 设计决策（已确认）
----------------------------------------------------------------

【A. 模式与配置】
- 默认模式：双模式并存，默认进入「练习模式」；可切换到「真实面试模式(live)」。
- 面试官人设(persona)：每场开始前由用户选择（友好引导 / 中性 / 高压快节奏）。
- 难度梯度：自适应（答得好就加难）+ 允许用户手动锁定/覆盖。
- 岗位侧重标签：加，且为可选（算法 / 概率统计 / ML / 市场直觉），影响抽样与
  提问偏好，不绑定具体公司。

【B. 对话进行中】
- 用户发废话/跑题/「我不知道」：顺势给引导/台阶，帮用户推进（两种模式都偏
  引导，不冷处理）。
- 用户反向提问/要求澄清：允许但有限——只回答合理的澄清性问题，越界提问会被
  婉拒，且不泄答案。
- live 模式的 Hint 与「查看参考答案」：均禁用（更真实）；普通练习模式两者均
  可使用/查看。
- 用户主动推进：可以主动「我答完了/跳过这题」，但跳过的题记为「未完成」。

【C. 反馈时机与展示】
- 单题反馈时机：真实面试模式整场结束后统一给反馈；练习模式每题作答后即时给。
- 维度评分展示：简洁分数 + 要点（总分 + 各维度数字 + 缺失要点 chips，不强制
  画雷达图，保持轻量）。
- 答错时面试官当场是否纠正：视模式而定——真实面试模式当场不纠正（留到结束
  反馈）；练习模式当场点出正确方向。
- 评分细则(rubric)：折叠可选（默认收起，用户想了解「为什么是这个分」时展开）。

【D. 面试结束界面】
- 显示内容：总分与逐题分（核心）。
- 结束操作：导出报告、存入练习记录、以相同配置重开一场。
- 历史趋势：要——记录每场分数与维度，做跨场进步趋势图。
- 导出格式：PDF。

【E. 语音与多模态】
- 读题 TTS：默认开启，可关闭。
- 语音作答轮转：手动按键提交（说完点一下提交，更可控；暂不做静音自动断句）。
- 服务端语音转写(Whisper)：暂不确定/后续再定——方案保留「Web Speech + 文本
  兜底」与「服务端转写」两条路径，先实现前者，Whisper 留作可选扩展点。
- 公式(KaTeX)与手写白板：暂不做，依赖现有文本与文件上传通道。

【F. 内容与稳健性】
- 行为面题库：扩充到 50+ 题（大题库，几乎不重题）。
- 题目去重/新鲜度：要——记录近期出过的题，抽样时降权。
- 会话持久化：要——live 多轮会话状态可持久化，刷新/关页后能恢复未完面试。
- 面试语言：每场可选（开始前为本场单独选中文/英文）。


----------------------------------------------------------------
1. 现状盘点（基于代码）
----------------------------------------------------------------

【前端 src/main.js】

- 面试类型 interviewTypeDefs：oa / technical / behavioral。其中 behavioral
  只有 4 道硬编码 STAR 题（behavioralInterviewProblems）。

- 配置项：类型、来源（题库 full / 上传 pdf）、作答方式（text / file /
  voice）、题数（1–12，getInterviewQuestionCount）、单题时间（1–60 分钟，
  getInterviewQuestionSeconds）、主题范围（renderInterviewCategoryPicker）。

- 主流程：startInterview → 生成题目 → 5 秒准备倒计时
  （startInterviewPrepCountdown）→ showInterviewQuestion（单题计时器）→
  submitInterviewAnswer → requestInterviewFeedback → 记录 → 下一题或
  completeInterview。

- submitInterviewAnswer 是一次性的：提交即评分并立即进入下一题，没有任何
  追问回合。startInterview 里 answerMode 实际被写死成 "chat"，而 setup
  暴露的 text/file/voice 选项并未真正驱动流程。

- 语音：SpeechRecognition（Web Speech API），continuous + interimResults，
  zh-CN / en-US，只是把识别文本灌进 els.interviewAnswer 文本框；没有读题
  TTS，没有对话轮转。

- 反馈：requestInterviewFeedback 调用 proxy 的 evaluate，返回单一 0–100 分
  + 四段（评价/遗漏/参考方向/下一步）；离线兜底 localInterviewFeedback。

- 其他：Hint（requestInterviewHint，不泄答案）、参考答案 reveal
  （revealInterviewAnswer）、收藏、PK 模式。

【LLM 代理 llm-proxy/server.mjs】

- 仅 3 个 task：evaluate / hint / generate_pdf_questions。

- evaluate：强制「单一总分」，刻意压缩到约 120–220 中文字 / 90–160 英文词；
  支持题目图片与作答附件（image / pdf）。


----------------------------------------------------------------
2. 核心差距（对照四个目标）
----------------------------------------------------------------

目标：多轮对话式追问
  现状：单题一次性提交即评分
  差距：完全缺失，无追问、无澄清、无动态深挖

目标：AI 反馈质量与深度
  现状：单一总分 + 四段紧凑文本
  差距：无分维度评分、无评分细则(rubric)、无与参考解法对比、无整场报告卡/趋势

目标：语音与多模态体验
  现状：语音仅作输入灌入文本框
  差距：无读题 TTS、无对话轮转、Web Speech 脆弱无兜底

目标：题目内容与覆盖
  现状：行为面 4 题硬编码
  差距：行为题库太薄、无岗位侧重、无难度分层与去重/新鲜度、缺心算/做市等
        交互题型


----------------------------------------------------------------
3. 优化方案
----------------------------------------------------------------

四大支柱，按优先级排序（多轮对话 > AI 反馈 > 语音多模态 > 内容覆盖）。每条
给出 目标 / 前端改动 / 代理改动 / 验收标准。已并入第 0 节决策。


================================================================
支柱 A：多轮对话式面试官（最高优先级）
================================================================

目标：每道题进入一段真实对话——面试官读题 → 候选人作答 → 面试官根据回答
追问 / 要求澄清 / 深挖边界条件 / 适度施压，在判定「问够了」后再进入下一题。

模式：双模式并存，默认「练习模式(practice)」；用户可切到「真实面试模式
(live)」。persona 每场开始前由用户选（友好引导 / 中性 / 高压快节奏）。

前端改动（src/main.js）：

- Setup 增加模式开关 practice / live（getInterviewMode()）、persona 选择、
  本场语言选择、可选「岗位侧重」标签、难度模式（自适应 / 手动锁定）。

- 新增对话状态机，替换 live 模式下的 submitInterviewAnswer 单次逻辑：
  * 每道题维护 turns（本题对话历史）、followupCount、maxFollowups
    （自适应，建议按难度 1–4）、interviewerSatisfied。
  * 提交回答 → 调用新代理 task converse → 返回结构化结果：
    { action: "followup" | "wrap", message, runningAssessment, coverage }。
    followup 时把面试官追问 append 到 transcript 并等待下一次作答；wrap 时
    进入下一题 showInterviewQuestion(next)。
  * 用户可主动「我答完了/跳过这题」→ 强制 wrap；跳过的题标记为「未完成」并
    计入结束报告。
  * 超时同样强制 wrap。

- 对话行为规则（注入 system prompt + 前端控制）：
  * 用户发废话/跑题/「我不知道」：顺势给引导/台阶帮其推进。
  * 用户反向提问：只回答合理澄清问题，越界提问婉拒，全程不泄答案。
  * live 模式禁用 Hint 与「查看参考答案」按钮；练习模式保留两者。
  * 难度自适应：答得好则后续题加难，反之降难；用户可手动锁定难度。

- UI：transcript 渲染区适配多轮气泡（面试官追问与候选人作答交替）。

代理改动（llm-proxy/server.mjs）：

- 新增 createInterviewConversation(payload)（task converse），挂到
  createInterviewReply 路由。

- system prompt：扮演面试官（含 persona）；输入题目 ground truth + 完整
  turns + 已用追问数 + 时间预算；要求只输出一条面试官发言（追问或收尾），
  并附 action 与累积评估（已覆盖点 / 还缺点）。返回紧凑 JSON。明确禁止泄答案。

- 收尾时复用/合并 evaluate 逻辑，产出该题最终分与维度分（见支柱 B）。

验收标准：live 模式一道概率题能产生 2–4 轮自然追问；不泄答案；跳过/超时
均能优雅收尾并标记未完成；整场结束有总评。


================================================================
支柱 B：AI 反馈质量与深度
================================================================

目标：从「一个总分 + 四段」升级为分维度评分 + 评分细则 + 整场报告卡，且保持
轻量（简洁分数 + 要点，不强制画雷达图）。

反馈时机：真实面试模式整场结束后统一给；练习模式每题作答后即时给。

代理改动（evaluate 与新 converse 收尾共用）：

- 输出结构化 JSON（不再是纯文本）：
  * overall（0–100）
  * dimensions：correctness / reasoning / communication / speed（各 0–5 +
    一句话）
  * missing：缺失要点清单（数组）
  * referenceDelta：候选人解法 vs 参考解法的关键差异（不复述参考答案）
  * nextStep：1–2 条可执行建议
- 保留紧凑约束（每段一句话），靠结构而非长度承载深度。

前端改动：

- normalizeInterviewFeedback 改为解析结构化 JSON（保留纯文本兜底，兼容旧
  evaluate）。

- 反馈渲染（轻量）：总分 + 各维度数字 + 「缺失要点」chips + 「你的解法 vs
  参考」对比块；不强制雷达图。可复用 animateInterviewScores 做数字动效。

- 答错处理：练习模式当场点出正确方向；真实面试模式当场不纠正，留到结束反馈。

- 评分细则(rubric)：折叠展示（默认收起，可展开看每维度 0–5 含义）。

- 整场报告卡（completeInterview 增强，结束界面，详见支柱 D / 第 0 节 D）。

验收标准：单题反馈含四维度分与缺失要点；练习/真实两种时机正确；旧版纯文本
反馈仍能正常显示（向后兼容）。


================================================================
支柱 C：语音与多模态体验
================================================================

目标：让「说」成为一等公民——面试官读题（TTS）+ 语音作答（手动提交）。
公式/手写暂不做。

前端改动：

- 读题 TTS：showInterviewQuestion / 面试官追问时用 speechSynthesis 朗读，
  默认开启、可关闭、可调语速，zh/en 跟随本场语言。

- 语音作答（手动提交）：保留并完善 SpeechRecognition，把识别文本填入作答框，
  用户说完点「提交」推进本轮（暂不做静音自动断句轮转）。保留录音状态可视化与
  「停止说话」。

- 兜底：Web Speech 仅 Chromium 稳定，非 Chrome 回退到文本输入并提示。
  （服务端转写 Whisper 作为可选扩展点保留，本期暂不实现 —— 见第 0 节 E。）

- 公式(KaTeX) / 手写白板：本期不做；如需表达推导，沿用现有文件/图片上传通道
  （proxy 端 createAttachmentInputPart 已支持 image）。

代理改动：

- 本期无强制改动；converse / evaluate 继续支持图片附件参与评估。
- （可选扩展）预留 transcribe task 接口设计，待 Whisper 决策确定后再实现。

验收标准：面试官能朗读题目且可关闭；语音作答能填入并手动提交；非 Chrome
浏览器语音降级到文本正常。


================================================================
支柱 D：题目内容与覆盖（通用/全覆盖）
================================================================

目标：保持全方向覆盖，补齐薄弱处，支持轻量定制。

- 行为面扩库：4 题 → 50+ 题的结构化行为题库（按主题：影响力/冲突/失败/抗压/
  领导力/为什么量化/为什么我们等），支持 STAR 追问（契合支柱 A 多轮）。

- 难度分层与自适应抽样：sampleInterviewQuestions 增加按 difficulty 配比
  （Easy/Medium/Hard）；live 模式答得好就加难，用户可手动锁定。

- 去重与新鲜度：记录近期出过的题，抽样时降权，避免连续几场重题。

- 交互题型增强：心算/做市/脑筋急转弯做成限时快问快答（与现有 Mental Math
  模块打通），而非纯文本评测。

- 轻量定制：setup 可选「岗位侧重」标签，影响抽样配比与面试官提问偏好；不绑定
  具体公司，保持通用。

验收标准：行为面 ≥50 题且能被追问；技术面含明显难度梯度；连续两场不大量
重题；可按岗位侧重调整题目分布。

4. 分阶段路线图
----------------------------------------------------------------

P0 地基
  内容：引入 interviewMode(practice/live) + persona + 本场语言 + 岗位侧重
        开关；evaluate 改结构化 JSON + 前端兼容解析；行为题库扩充(50+)
  价值：立刻提升反馈深度与可配置性，且不破坏现状
  改动量：中   风险：低（向后兼容）

P1 多轮对话
  内容：新增代理 converse task + 前端对话状态机 + persona 注入 + 废话引导/
        反向提问规则 + live 禁用 Hint/答案 + 跳过(记未完成)/超时收尾 +
        难度自适应
  价值：最高优先级，体验质变
  改动量：大   风险：中（prompt 调优、状态机边界）

P2 反馈报告与结束界面
  内容：维度评分(简洁分数+要点) + rubric 折叠 + 结束界面(总分&逐题分 +
        导出PDF + 存入练习记录 + 重开同配置) + 跨场趋势图 + 答错纠正按模式
  价值：让练习「可复盘、可追踪进步」
  改动量：中大  风险：低

P3 语音
  内容：读题 TTS(默认开可关) + 语音作答手动提交完善 + 非 Chrome 文本兜底
  价值：沉浸式真实感
  改动量：中   风险：中（浏览器兼容）

P4 内容与稳健性收尾
  内容：难度分层/去重新鲜度 + 心算做市交互题 + 会话持久化与刷新恢复
  价值：覆盖补齐、体验稳健
  改动量：中   风险：低

建议落地顺序：P0 → P1 → P2 → P3 → P4。P0+P1 完成后即获得「多轮对话 +
更深反馈」的核心升级。
（Whisper 服务端转写为待定扩展点，可在 P3 之后按需插入。）


----------------------------------------------------------------
5. 技术风险与注意点
----------------------------------------------------------------

- 向后兼容：converse/结构化 evaluate 上线时，前端解析必须保留旧纯文本兜底
  （localInterviewFeedback 路径不变），避免 proxy 未更新时白屏。

- 成本与延迟：多轮对话显著增加 LLM 调用次数；对追问数设上限（按难度 1–4）、
  用更快模型档位（沿用 DEFAULT_MODEL 可配），并对 converse 输出做 token 预算。

- Prompt 越权：面试官追问时容易「不小心给答案」；system prompt 明确禁止泄
  答案，收尾前不输出参考解法；live 模式前端也禁用 Hint/答案按钮双保险。

- 语音兼容：Web Speech 仅 Chromium 稳定；非 Chrome 回退文本输入。Whisper
  兜底为待定扩展点（有 API 成本）。

- 状态机与持久化：超时、用户中途跳题/切题、刷新——live 多轮会话状态需持久化
  并可恢复未完面试（第 0 节 F）。

- i18n：所有新文案需同时进 src/i18n.js 的 zh/en；面试语言每场可选，需与全局
  语言解耦。

6. 验收与测试
----------------------------------------------------------------

- 静态检查：node --check src/main.js、node --check llm-proxy/server.mjs、

- 反馈正确性回归：对同一题构造「好答案 / 漏要点 / 答错」三类输入，核对维度分
  与缺失要点是否合理；核对练习即时 vs 真实结束统一两种时机。

- 兼容性回归：proxy 仍返回旧纯文本时前端不报错；非 Chrome 浏览器语音降级
  到文本正常。
